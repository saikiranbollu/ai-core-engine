## 2. System Purpose and Scope

### 2.1 What AICE Is

AICE is a **knowledge-graph-backed retrieval and context assembly server** that serves structured engineering knowledge to LLM-based Domain Assistants. It exposes 56 MCP tools across 13 categories, backed by a Hybrid RAG engine combining Neo4j graph traversal with Qdrant vector similarity search.

AICE provides: API function signatures, register maps, requirements traceability chains, compliance rules, dependency graphs, initialization ordering, test results, and MISRA/AUTOSAR compliance data.

### 2.2 What AICE Is NOT

- **AICE is NOT an LLM.** It does not generate text, code, or decisions. It retrieves and assembles context that LLMs in Domain Assistants consume.
- **AICE is NOT an in-vehicle AI system.** It operates exclusively in the development environment. No AICE component runs on vehicle hardware or influences vehicle behavior at runtime.
- **AICE is NOT a safety system.** It is a development productivity tool. The safety-critical outputs are the responsibility of the Domain Assistants and, ultimately, the human engineers who review and approve them.

### 2.3 Intended Use

AICE is intended to accelerate embedded software development workflows by providing accurate, contextual engineering knowledge to AI-assisted tools. The outputs of AICE-backed Domain Assistants include generated code, test specifications, requirement reviews, architecture analyses, and safety assessments — all subject to mandatory human review before integration into production software.

### 2.4 Unintended and Prohibited Uses

| Category | Description |
|----------|-------------|
| **Prohibited** | Direct deployment of AI-generated code into safety-critical paths without human review |
| **Prohibited** | Use of AICE outputs as formal safety evidence without independent verification |
| **Prohibited** | Autonomous code commit without Review Gate completion |
| **Prohibited** | Processing of personal data, biometric data, or employee performance data |
| **Unintended** | Use as a replacement for domain expertise in safety analysis (FMEA, FTA, DFA) |
| **Unintended** | Use for security-sensitive code paths (cryptographic implementations, access control) without security expert review |

## 3. EU AI Act Risk Classification

### 3.1 Classification Rationale

Under the EU AI Act (Regulation EU 2024/1689), AICE is classified as follows:

| Classification Aspect | Assessment |
|----------------------|------------|
| **Annex I (product-embedded)** | **Not applicable.** AICE does not operate within a vehicle or regulated product. |
| **Annex III (standalone high-risk)** | **Not applicable.** AICE does not fall into any of the eight high-risk areas (biometric, education, employment, etc.). |
| **Risk level** | **Limited risk** (transparency obligations apply) |
| **Rationale** | AICE is a developer tool whose outputs undergo mandatory human review. It does not autonomously affect safety-critical systems. The code it helps produce enters regulated products, but AICE itself is a development-time tool, analogous to a compiler, static analyzer, or IDE with AI assistance. |

### 3.2 Downstream Risk Cascade

Although AICE itself is limited-risk, its outputs feed into high-risk contexts:

- **Code generated by CIA** enters AUTOSAR MCAL drivers classified up to ASIL-D
- **Test specifications from GEST** verify safety-critical functionality
- **Safety analyses from SAVA/SAAN** contribute to ISO 26262 safety cases

This downstream cascade means AICE must maintain governance controls proportional to the highest ASIL level of the software it supports, even though AICE itself is not classified as high-risk. The Human Review Gate is the primary control mechanism for this cascade.

### 3.3 Transparency Obligations

Under EU AI Act Article 50, the following transparency measures are implemented:

| Obligation | Implementation |
|-----------|---------------|
| Users informed they interact with AI | Domain Assistants identify themselves as AI-assisted tools in all UIs |
| AI-generated content marked | All AI-generated code includes `/* AI-GENERATED — Review Required */` header comments |
| Provenance recorded | Every AI-assisted output is logged with session ID, tool chain, and context sources in PostgreSQL audit trail |

## 4. Technical Architecture Summary

### 4.1 Component Stack

| Component | Technology | Role |
|-----------|-----------|------|
| Knowledge Graph | Neo4j 5.26 | Structured engineering data and relationships |
| Vector Store | Qdrant 1.12 | 384-dim semantic embeddings (all-MiniLM-L6-v2) |
| Session/Cache | Redis 7 | Working memory, LRU cache, session TTL |
| Audit Store | PostgreSQL 16 | 7-table audit schema (ASPICE-compliant) |
| Authorization | Cerbos PDP | 3-tier RBAC (public/developer/admin) |
| Metrics | Prometheus + Grafana | 11 metric types, 15s scrape, 10-panel dashboard |
| LLM Proxy | GPT4IFX | Infineon internal LLM gateway (RLM planning, PDF extraction) |
| Framework | FastMCP + Uvicorn | MCP server with ASGI middleware |

### 4.2 Data Flow

```
User Request → DA (LLM) → MCP Call → AICE Server
    → Auth Check (Cerbos) → Tool Handler
    → Hybrid Search (Neo4j + Qdrant) → Context Assembly
    → Response to DA → DA generates output → Confidence Score
    → Review Gate (AUTO/QUICK/FULL) → Human Review
    → Feedback → PatternStore (learning loop)
    → Audit Log (PostgreSQL)
```

### 4.3 LLM Dependencies

| LLM Usage Point | Model | Purpose | Risk |
|----------------|-------|---------|------|
| RLM Orchestrator | GPT4IFX | Sub-query planning for complex retrievals | Medium — plans are validated before execution |
| PDF Extraction | GPT4IFX | Table/diagram extraction from HW docs | Low — extracted data verified against source |
| Domain Assistants | GPT4IFX / GitHub Copilot | Code/test/review generation | High — outputs enter safety-critical paths |

AICE's own LLM usage (RLM planning, PDF extraction) is limited to retrieval optimization. The high-risk LLM usage occurs in Domain Assistants, which are separate systems.

## 5. Data Governance

### 5.1 Data Sources

| Source | Type | Sensitivity | Ingestion Method |
|--------|------|-------------|-----------------|
| iLLD source code | C/H files | Infineon confidential | Tree-sitter AST parsing |
| MCAL requirements | Jama Connect exports | Infineon confidential | Jama XML/JSON parsers |
| Hardware specifications | PDF documents | Infineon confidential | LLM-assisted PDF extraction |
| AUTOSAR SWS documents | PDF/XML | Public standard (licensed) | RST/XML parsers |
| Test results | JUnit XML, Polyspace CSV | Internal | Result processors |
| Enterprise Architect models | XMI export | Infineon confidential | EA parser |

### 5.2 Data Lifecycle

| Phase | Controls |
|-------|----------|
| **Ingestion** | Admin-tier authorization required; `ingestion_jobs` table tracks source, parser, timestamp, node/relationship counts |
| **Storage** | Dual workspace isolation (illd/mcal); per-module NodeSet anchors prevent cross-module data bleed; separate Neo4j databases per workspace |
| **Access** | Cerbos RBAC; API keys mapped to principals; tool-tier enforcement |
| **Retention** | KG data persists until explicit re-ingestion; audit logs retained per organizational policy; Ephemeral Sandbox data expires with session TTL |
| **Deletion** | Module-level data can be cleared via admin tools; no personal data processed |

### 5.3 Personal Data Assessment

AICE does **not** process personal data as defined by GDPR Article 4(1). All data in the knowledge graph is technical engineering data (source code, specifications, requirements, register maps). No employee names, performance data, or behavioral data enters the system. API key principals are system identifiers (e.g., `cia_assistant`), not personal identifiers.

## 6. Quality and Safety Controls

### 6.1 Confidence Scoring (Deterministic)

The Review Gate uses a deterministic formula (not LLM-based):

```
Score = clamp(50 + Σ(quality_signals) - Σ(risk_signals), 0, 100)

Quality: has_kg_context(+30), high_relevance(+20), has_dependency_order(+20),
         has_proven_patterns(+15), format_correct(+10), misra_compliant(+10),
         similar_approved(+5)

Risk:    missing_requirements(-30), low_relevance(-20), compliance_warnings(-20),
         novel_pattern(-15), is_safety_critical(-15), complex_logic(-10)

Routing: AUTO (≥80) | QUICK (50-79) | FULL (<50)
```

### 6.2 Review Gate Enforcement

| ASIL Level | Minimum Review | Rationale |
|-----------|---------------|-----------|
| QM (non-safety) | AUTO permitted | Low risk, spot-check sufficient |
| ASIL-A | QUICK minimum | Safety awareness required |
| ASIL-B | FULL mandatory | Safety-critical, expert review |
| ASIL-C/D | FULL mandatory + independent reviewer | Highest integrity level |

**Governance rule**: The `is_safety_critical` signal must be set to `true` for any module with ASIL-B or above classification. This automatically reduces the confidence score by 15 points, making AUTO routing highly unlikely.

### 6.3 Feedback and Learning Loop

Approved patterns are stored in Neo4j PatternStore and Qdrant PatternIndex. This creates a positive feedback loop where previously approved patterns boost future confidence scores. Controls:

- Only `APPROVE` and `APPROVE_WITH_EDITS` verdicts generate patterns
- `APPROVE_WITH_EDITS` patterns are stored with reduced confidence (0.75)
- `REJECT` verdicts are stored as failure patterns in PostgreSQL for analysis
- Pattern extraction is auditable and reversible

### 6.4 Observability

| Metric Category | Implementation |
|----------------|---------------|
| Tool usage | `tool_requests_total`, `tool_request_duration` per tool per DA |
| Search quality | `search_requests_total`, `search_duration` |
| Cache efficiency | `cache_requests_total` (hit/miss/eviction) |
| Review activity | `review_routing_total` (AUTO/QUICK/FULL distribution) |
| System health | `backend_up` gauge for all 7 services |

## 7. Known Limitations and Failure Modes

### 7.1 Technical Limitations

| Limitation | Impact | Mitigation |
|-----------|--------|-----------|
| KG coverage is not complete for all modules | Missing context leads to lower-quality retrievals | `find_coverage_gaps` tool reports missing links; `missing_requirements` signal reduces confidence score |
| Embedding model (MiniLM-L6-v2) is general-purpose | May miss domain-specific semantic similarities | Graph-RAG (structural queries) compensates; ~75% of AICE queries are graph-structural |
| RLM planning depends on LLM | Sub-query plans may be suboptimal | Plans are validated and bounded (max 6 sub-queries); fallback to single hybrid search |
| Semantic cache similarity threshold (0.85) | Near-miss queries may return stale results | LRU cache uses exact match; cache invalidation on KG updates planned |

### 7.2 Failure Modes

| Failure | Detection | Response |
|---------|-----------|----------|
| Neo4j unavailable | Health check, `backend_up` gauge | Graceful degradation — vector-only search continues |
| Qdrant unavailable | Health check | Graceful degradation — graph-only search continues |
| Redis unavailable | Health check | No cache — all queries go to full RAG pipeline |
| PostgreSQL unavailable | Non-blocking writes | Audit logs lost during outage (logged to stderr); no data corruption |
| GPT4IFX unavailable | Token refresh failure | RLM disabled — falls back to single hybrid search; PDF extraction fails |
| Cerbos unavailable | Sidecar health check | Fallback to local tier-check from `tool_tiers.py` |

### 7.3 Bias and Fairness Considerations

In the automotive embedded software domain, "bias" manifests as:

| Bias Type | Risk | Assessment |
|-----------|------|-----------|
| **Module coverage bias** | Some modules have richer KG data than others | Measurable via `get_graph_statistics`; documented per module |
| **Pattern approval bias** | Frequently-used patterns get more approvals, reinforcing themselves | Mitigated by `novel_pattern` signal and periodic pattern review |
| **Workspace bias** | MCAL workspace may have different coverage than iLLD | Separate statistics per workspace; workspace-specific quality metrics |
| **Source bias** | Some parsers extract more completely than others | Parser-level quality metrics tracked in ingestion pipeline |

## 8. Human Oversight

### 8.1 Oversight Mechanisms

| Mechanism | Description |
|-----------|-------------|
| **Review Gate** | Every AI-assisted output is scored and routed to appropriate review level |
| **Confidence Breakdown** | Engineers see which signals contributed to the score — fully explainable |
| **Override Capability** | Any review routing can be overridden (e.g., escalate AUTO to FULL) with documented reason |
| **Feedback Loop** | Engineer decisions (approve/reject/edit) are recorded and improve future scoring |
| **Audit Trail** | Complete PostgreSQL log of every tool invocation, every review decision, every feedback event |

### 8.2 Human Authority

The human engineer retains full authority to:

- Override any AI-generated output
- Escalate any review to a higher level
- Reject any AI-generated artifact for any reason
- Modify AI-generated code before approval
- Disable AI assistance for specific tasks
- Request regeneration with different parameters

No AICE tool or Domain Assistant may autonomously commit code, merge branches, release artifacts, or modify production systems.

## 9. Incident Response

### 9.1 AI-Related Incident Classification

| Severity | Definition | Example | Response |
|----------|-----------|---------|----------|
| **Critical** | AI output caused or could cause a safety hazard | Generated code with incorrect register access that could damage hardware | Immediate recall of all outputs from same session; root cause analysis; KG data audit |
| **High** | AI output contains significant technical errors | Missing DET error handling in generated driver code | Block AUTO approval for affected module; investigate KG coverage |
| **Medium** | AI output has quality issues | Suboptimal initialization sequence; missing comments | Feedback as REJECT; pattern update; process review |
| **Low** | Minor formatting or style issues | Incorrect indentation; non-standard naming | Feedback as APPROVE_WITH_EDITS; style rule update |

### 9.2 Incident Documentation

Every AI-related incident must be documented with:

- Session ID and tool invocation audit trail
- The context that was assembled (from `response_archive`)
- The confidence score and signal breakdown
- The review decision and reviewer identity
- Root cause analysis (KG data issue, retrieval failure, LLM hallucination, prompt issue)
- Corrective action taken
