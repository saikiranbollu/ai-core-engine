## 2. Purpose and Scope

### 2.1 What AICE Is

AICE is a **knowledge-graph-backed retrieval and context assembly server** that serves structured engineering knowledge to LLM-based Domain Assistants. It exposes 50+ MCP tools across 13 categories, backed by a Hybrid RAG engine combining Neo4j graph traversal with Qdrant vector similarity search. AICE retrieves and assembles context; the LLMs (Copilot Enterprise primarily; GPT4IFX for specialized paths) perform generation.

### 2.2 What AICE Is NOT

- **Not an LLM.** AICE retrieves and assembles context that LLMs in Domain Assistants consume.
- **Not an in-vehicle AI system.** AICE operates exclusively in the development environment.
- **Not itself a safety-critical system in the ISO 26262 Part 4 sense.** It is a development productivity tool governed by ISO 26262 Part 8 Clause 11 (tool qualification).

### 2.3 Intended Use

Accelerate embedded software development workflows by providing accurate, contextual engineering knowledge to AI-assisted tools. Outputs of AICE-backed Domain Assistants — generated code, test specifications, requirement reviews, architecture analyses, safety assessments — are **delivered to Infineon customers (German OEMs and Tier-1s) as part of productive MCAL/iLLD software deliveries, after mandatory human review**. The AI itself does not ship.

### 2.4 Prohibited Uses

| Category | Description |
|---|---|
| Direct AI commit | Direct integration of AI-generated code into productive baseline without Human Review Gate completion |
| Safety evidence | Use of AICE outputs as formal safety evidence without independent expert verification |
| Autonomous commit | All commits are under the reviewing engineer's identity |
| Personal data | Processing of personal data, biometric data, employee performance data |
| Non-compliant routing | Customer-NDA content routed via any LLM endpoint not contractually approved (see §4.3) |
| AI for safety analysis | FMEA / FTA / DFA / HAZOP outputs used directly without expert review |
| AI for security-sensitive code | Crypto implementations or access-control logic without security expert review |
| AI for customer-facing artifacts | Manuals or datasheets without marketing/legal review |

- **Code** generated by CIA / CTA → AUTOSAR MCAL drivers up to ASIL-D, German OEMs

## 4. Technical Architecture

### 4.1 Component Stack

| Component | Technology | Role |
|---|---|---|
| Knowledge Graph (persistent) | Neo4j 5.26 | Structured engineering data and relationships |
| Vector Store (persistent) | Qdrant 1.12 | Semantic embeddings (384-dim all-MiniLM-L6-v2) |
| **Sandbox graph (ephemeral)** | NetworkX in-memory | Session-scoped temporary KG for user-uploaded EA models / specs |
| **Sandbox vectors (ephemeral)** | ChromaDB in-memory | Session-scoped vectors; physically released on session end |
| Session/Cache | Redis 7 | Working memory, LRU cache, session TTL |
| Audit Store | PostgreSQL 16 | 7-table audit schema; WORM layer for compliance evidence |
| Authorization | Cerbos PDP | 3-tier RBAC; per-DA principals |
| Metrics | Prometheus + Grafana | 11 metric types, governance dashboard |
| **Primary LLM** | **GitHub Copilot Enterprise** (Microsoft cloud, EU Data Boundary, no-training) | DA generation across all zones |
| **Specialized LLM** | **GPT4IFX** (Infineon on-prem) | PDF extraction; RLM planning; fallback |
| Framework | FastMCP + Uvicorn | MCP server with ASGI middleware |
| **EA Architecture Source** | **`ifxpyarch` MCP server** (separate process, MCP transport) | Direct QEAX file parsing for MCAL architecture model — replaces PDF-based architecture ingestion |

### 4.2 Data Flow

```
User Request → DA (Copilot Enterprise) → MCP Call → AICE Server
    → Auth Check (Cerbos) → Tool Handler
    → Hybrid Search:
        Shallow tools → Sandbox NetworkX/ChromaDB if active session
        Deep tools  → Neo4j + Qdrant (production), patched/injected with sandbox overrides
    → Context Assembly → Response to DA
    → DA generates output via Copilot (or GPT4IFX for specialized paths)
    → AI-transparency markers injected (code header, git trailer)
    → Confidence Score (deterministic formula)
    → ASIL check → Review Gate (AUTO/QUICK/FULL with ASIL-D hard-gate)
    → Human Review → Static analysis pipeline (Zone A: full pipeline mandatory)
    → Commit under reviewer identity with provenance trailer
    → Feedback → PatternStore (subject to sandbox firewall)
    → Audit Log (PostgreSQL + WORM archive)
```

### 4.3 LLM Routing Policy

**Primary policy: Copilot Enterprise for all DAs across all zones, including Customer-NDA content**, subject to contractual preconditions:

| Required Copilot Enterprise control | Status |
|---|---|
| **EU Data Boundary** for prompt/completion processing | Required — verified in MS commercial agreement |
| **No-training assurance** — Microsoft contractually does not use Infineon prompts/completions to train models | Required — verified in MS commercial agreement |
| **Audit logs accessible** to Infineon (prompt/completion metadata) | Required — verified in MS commercial agreement |
| **"Block matching public code"** setting org-wide enforced | Required — see AICE-GOV-009 §3.1 |

| **Tenant isolation** confirmed | Required |

**If any precondition ceases to hold for a specific data class** (customer NDA contract requires data residency stricter than EU Data Boundary, Microsoft changes terms), routing for that data class falls back to GPT4IFX.

| Data Class | Examples | Copilot Enterprise (with §4.3 preconditions) | GPT4IFX (specialized) |
|---|---|---|---|
| Public — Infineon-published | Published iLLD on github.com/Infineon, public TC3xx datasheet | ✅ allowed | Optional |

**GPT4IFX specialized paths (always GPT4IFX, regardless of Copilot status):**
- PDF extraction (HW spec PDFs, errata PDFs)
- RLM (Recursive Language Model) planning for complex multi-step retrieval
- Fallback when Copilot Enterprise unavailable
- DAs where the contractual posture for Copilot is uncertain

Classification is deterministic (regex + keyword + ontology tag lookup) before prompt construction. Routing enforced via Cerbos pre-LLM policy.

### 4.5 ifxpyarch MCP Integration

For MCAL architecture (and increasingly for any EA-modelled component), AICE no longer relies on PDF-based architecture ingestion. The `ifxpyarch` MCP server (separate process, stdio transport, install via `uv tool install ifxpyarch-mcp`) parses `.qeax` files (Enterprise Architect SQLite exports) directly and exposes 20+ MCP tools (`process_request` gateway + raw DB tools + metamodel introspection).

**Two integration paths:**

1. **Persistent KG ingestion path:** `IngestionPipeline/Parsers/ea_parser.py` reuses the same EA parsing logic to ingest QEAX content into the persistent Neo4j (workspace-scoped, admin-tier, signed ingestion bundles).
2. **Sandbox path (ephemeral):** A user uploads a `.qeax` file via `sandbox_upload`. The same parser produces NetworkX nodes/edges in the per-session sandbox. Subsequent retrievals layer this on top of the persistent Neo4j (deep tools) or query sandbox-only (shallow tools).

**Governance benefit of the QEAX path:** structured ingestion (deterministic schema, no LLM-assisted parsing) replaces the higher-risk PDF-extraction path. Prompt-injection surface from natural-language documents is reduced for all architecture-related work routed through QEAX.

## 5. Data Governance

### 5.1 Data Sources

| Source | Type | Sensitivity | Ingestion Method |
|---|---|---|---|
| **Production EA models (.qeax) for MCAL** | Structured architecture model | Infineon confidential / Customer-NDA | EA parser via `ifxpyarch` (production ingestion path) |
| iLLD pre-release source | C/H files | Infineon confidential | Tree-sitter AST parsing |
| iLLD published release | C/H files | Public — Infineon Free License | Tree-sitter AST parsing |
| MCAL requirements (Jama) | Requirements (SHRQ, PRQ, VS, VR) | Infineon confidential / Customer NDA | Jama XML/JSON parsers |
| Hardware specifications | PDF | Infineon confidential | LLM-assisted PDF extraction (via GPT4IFX, scoped) |
| AUTOSAR SWS documents | PDF/XML | Licensed (AUTOSAR) | RST/XML parsers |
| Test results | JUnit / Polyspace | Internal | Result processors |
| Zephyr upstream | C/H, device tree | Public — Apache-2.0 | Tree-sitter + devicetree parser |
| NuttX upstream | C/H, kconfig | Public — BSD-3 | Tree-sitter + config parser |

### 5.2 Data Lifecycle

| Phase | Controls |
|---|---|
| Persistent ingestion | Admin-tier authorization; `ingestion_jobs` tracks source/parser/timestamp; signed ingestion bundles (manifest SHA256); content-type guard; PII scrubber |
| **Sandbox ingestion (per-session)** | User-tier upload via DA session; deterministic data-class classifier on upload; content-type guard + PII scrubber; audit logged. See AICE-GOV-010 |
| Storage | Workspace-isolated Neo4j databases (illd / mcal / mcal_customer_X); per-module NodeSet anchors; per-customer sub-workspace for NDA. Sandbox content lives in NetworkX + ChromaDB in-memory only |
| Access | Cerbos RBAC; per-DA principals; tier enforcement; no cross-workspace queries; sandbox access scoped to originating session |
| Retention | Persistent KG until re-ingestion; audit logs ≥ product lifetime + 10 years; sandbox session state expires at TTL (default 1h); WORM snapshots of sandbox-grounded generations retained per zone |
| Deletion | Module-level via admin tools; sandbox session content physically released from RAM at TTL; WORM snapshots persist for reproducibility |

### 5.3 Personal Data Assessment

AICE does not intentionally process personal data per GDPR Art. 4(1) or DPDP Act 2023. Data in the KG is technical engineering data. API key principals are system identifiers (e.g., `cia_assistant`), not personal.

### 5.4 PII Scrubber

Defensive control on all paths into LLM prompts:
- Patterns scrubbed: personal names, email addresses, phone numbers, postal addresses, IP addresses identifying individuals
- Preserved: role tokens substituted back in UI
- Logged without PII content
- Test canaries verify scrubber operation
- Failure mode: prompt path blocks (fail-closed)
- Scope: prompt path + sandbox-derived chunks + KG-derived chunks

### 5.5 Sandbox & Ephemeral Data

The Ephemeral Sandbox is a per-session feature (status: implemented Sprint 4-5) that:
- Accepts user-uploaded content (QEAX models via ifxpyarch parser, plus PDFs, customer specs, meeting notes, compiler logs)
- Stores data in NetworkX (graph) + ChromaDB in-memory (vectors) — physically released on session end
- Layers on top of persistent Neo4j+Qdrant via `HybridGraphService`: shallow tools query sandbox only; deep tools query production with `_patched` (sandbox overrides) and `_injected` (sandbox-only nodes) flags

**Critical governance implications** (full specification in AICE-GOV-010):

| Concern | Control |
|---|---|
| Sandbox-grounded generations cannot be reproduced after session TTL | C1: WORM snapshot at generation time (mandatory) |
| Sandbox content can override production KG silently | New: `_patched`/`_injected` flags must surface to reviewers |
| `ephemeral_boost` (default +0.05) gives sandbox priority over production | Must be set to 0 for ASIL-D sessions |
| Sandbox content can poison the pattern store | C7: pattern store firewall blocks pattern creation from sandbox-grounded generations |
| Customer-NDA content uploaded mid-session must trigger routing change | C3: data-class inheritance on upload |

## 6. Quality and Safety Controls

### 6.1 Confidence Scoring (Deterministic)

```
Score = clamp(50 + Σ(quality_signals) − Σ(risk_signals), 0, 100)

Quality: has_kg_context(+30), high_relevance(+20), has_dependency_order(+20),
         has_proven_patterns(+15), format_correct(+10), misra_compliant(+10),
         similar_approved(+5)
Risk:    missing_requirements(−30), low_relevance(−20), compliance_warnings(−20),
         novel_pattern(−15), is_safety_critical(−15), complex_logic(−10)
Routing: AUTO (≥80) | QUICK (50–79) | FULL (<50)
```

### 6.2 Review Gate Enforcement

**Zone A (mcal — productive, up to ASIL-D):**

| ASIL Level | Minimum Review | Override Rule |
|---|---|---|
| QM | AUTO permitted | None |
| ASIL-A | QUICK | None |
| ASIL-B | FULL mandatory | Cannot downgrade via score |
| ASIL-C | FULL + independent | Cannot downgrade via score |
| **ASIL-D** | **FULL + independent + Safety Manager sign-off** | **Hard-gated.** `asil=D` tag forces routing |

**ASIL-D + sandbox-grounded special rule:** generations grounded on sandbox content force FULL + independent + Safety Manager regardless of score. Reviewer must verify sandbox content against authoritative source; `review_evidence.sandbox_source_verified` recorded.

**Zone B (illd — no ASIL claim):**

| Activity | Default review |
|---|---|
| Driver code, headers | QUICK |
| Examples, demos | AUTO permitted |
| High-reuse references (crypto, secure boot, MPU setup, interrupt vectors) | FULL recommended |

**Zone C (foss-bsp — Zephyr / NuttX BSP):**

| Activity | Default review |
|---|---|
| Device tree, kconfig | QUICK |
| Driver glue (UART, SPI, I2C, GPIO) | QUICK |
| FFI-critical (MPU, interrupt vectors, startup, hypervisor, secure boot) | FULL + dual reviewer |

### 6.4 Feedback and Learning Loop

Approved patterns stored in Neo4j PatternStore + Qdrant PatternIndex with controls:
- Only `APPROVE` and `APPROVE_WITH_EDITS` verdicts generate patterns
- `APPROVE_WITH_EDITS` reduces confidence (0.75)
- `REJECT` stores failure patterns
- **Pattern store firewall:** patterns from sandbox-grounded generations blocked unless sandbox content is formally promoted to persistent KG (AICE-GOV-010 §C7)
- Pattern expiration: 12-month review, 24-month auto-retire

## 7. Known Limitations and Failure Modes

### 7.1 Technical Limitations

| Limitation | Impact | Mitigation |
|---|---|---|
| KG coverage incomplete for some modules | Lower-quality retrievals | `find_coverage_gaps`; `missing_requirements` signal |
| Embedding model is general-purpose (MiniLM-L6-v2) | May miss domain-specific similarities | Graph-RAG compensates; ~75% queries are graph-structural |
| RLM planning depends on LLM | Plans may be suboptimal | Plans bounded (max 6 sub-queries); fallback to single hybrid search |
| Semantic cache similarity threshold (0.85) | Near-miss queries may return stale results | LRU exact match; cache invalidation on KG updates |
| LLM non-determinism | Same prompt may yield different outputs | Low temperature (0.2); prompt template versioning; static analysis catches deviations |
| Sandbox `ephemeral_boost` (+0.05) prioritizes sandbox over production | Un-vetted content can outrank reviewed KG | Set boost = 0 for ASIL-D sessions (AICE-GOV-010) |

### 7.2 Failure Modes

| Failure | Detection | Response |
|---|---|---|
| Neo4j unavailable | Health check; `backend_up` | Vector-only search continues |
| Qdrant unavailable | Health check | Graph-only search continues |
| Redis unavailable | Health check | No cache; queries hit full RAG |
| PostgreSQL unavailable | Non-blocking writes | Audit buffered to stderr; replay on recovery |
| **Copilot Enterprise unavailable** | API health | DA generation falls back to GPT4IFX where data class permits; otherwise blocks |
| GPT4IFX unavailable | Token refresh failure | RLM disabled; PDF-extraction blocked; specialized DAs block |
| Cerbos unavailable | Sidecar health | **Fail-closed** |
| PII scrubber malfunction | Test canaries | Prompt path blocked |
| Review gate ASIL override bypass attempt | Cerbos + evaluate_confidence logic | Logged as governance incident; blocked |
| Sandbox WORM storage unavailable | Health check | **Fail-closed** for sandbox-grounded generation |
| Sandbox content-type guard detects injection | Scanner verdict | Upload rejected; LOW governance incident |
| Sandbox data-class classifier ambiguous | Confidence below threshold | Upload accepted at most-restrictive class; flagged |
| Cross-session ChromaDB leak attempt | `cache_shareable=false` flag check | Cache miss forced; logged |
| Pattern store firewall refusal | Sandbox-grounded pattern creation blocked | LOW severity log; expected |
| **Microsoft Copilot Enterprise contract change** | Legal monitoring | Re-route affected data classes to GPT4IFX; update §4.3 |

### 7.3 Bias and Fairness

In dev-tool context, "bias" means *technical bias* not demographic:

| Bias Type | Risk | Mitigation |
|---|---|---|
| Module coverage bias | Some modules richer in KG | `get_graph_statistics`; per-module documented |
| Pattern approval bias | Frequently-used patterns reinforce themselves | `novel_pattern` signal; pattern expiration |
| Workspace bias | MCAL ≠ iLLD coverage | Separate stats per workspace |
| **Copilot training-data bias** | Trained on public GitHub; may favor non-AUTOSAR idioms | Mitigated by RAG grounding + static analysis + approved pattern store |

## 8. Human Oversight

### 8.1 Mechanisms

| Mechanism | Description |
|---|---|
| Review Gate | Every AI-assisted output scored and routed; ASIL-gated override for ASIL-D |
| Confidence Breakdown | Engineers see signal contributions; fully explainable |
| Override Capability | Routing can be escalated; downgrades below policy minimum blocked |
| Feedback Loop | Engineer decisions recorded; improve future scoring |
| Audit Trail | PostgreSQL log + WORM layer |
| Incident Response | Formal runbook (AICE-GOV-005); field-failure → AI-lineage traceback |
| Sandbox Diff | `sandbox_diff` tool surfaces what sandbox changed vs production; mandatory for ASIL-D sandbox-grounded acceptance |

### 8.2 Human Authority

The human engineer retains full authority to override any AI-generated output, escalate any review, reject any artifact, modify AI-generated code before approval, disable AI assistance, or request regeneration. No AICE tool or DA may autonomously commit code, merge branches, release artifacts, or modify production systems.

## 9. Incident Response

See AICE-GOV-005 for full procedure. Summary:

### 9.1 Classification

| Severity | Definition | Response Time |
|---|---|---|
| CRITICAL | Could cause safety hazard in customer-delivered field; or breaks Art. 73 threshold | 2h triage / 48h resolution |
| HIGH | Significant errors past review, caught pre-ship | 4h / 5 business days |
| MEDIUM | Quality issues caught in review | 1 business day / 10 days |
| LOW | Minor formatting/style | 3 days / 20 days |

### 9.2 Field-Failure → AI-Lineage Traceback (Phase 0)

Customer reports field failure → query `response_archive` by commit SHA → retrieve provenance → if sandbox refs present, retrieve WORM snapshots and verify integrity → assess AI-causation → escalate per §9.1.

### 9.3 EU AI Act Art. 73 Reporting

If AI-assisted artifact contributes to a serious field incident in a high-risk AI system (vehicle), reporting deadlines: 2 days (death/serious harm); 15 days (widespread/serious). Provenance chain is the primary investigative instrument.
