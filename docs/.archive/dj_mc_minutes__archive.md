Mothy Gobbilla explains the DJ environment used by the AT test engineering team, highlighting the integration of various documentation sources such as Confluence, file shares, Jira, and Windshield content containers. Source code repositories like GitLab are planned to be included soon. The environment is identified by a unique ID used for configuration in the MCP server.
The team discusses how the MCP server interacts with DocJockey to process queries. Authentication is currently handled via GitHub tokens decoded to identify users and their access rights. Plans exist to move towards IoT-based authentication to remove dependency on GitHub tokens. The MCP server currently returns retriever results, with plans to support direct answer generation in the future.
Mothy Gobbilla outlines the architecture supporting concurrency and scalability for MCP server deployments, including auto-scaling mechanisms to handle increased request loads. The system is deployed on HICP and follows standard asynchronous practices to ensure reliable performance under concurrent access.
Traceability at a granular level is a critical requirement, especially for automotive standards compliance. The team discusses the limitations of DocJockey's general offering in providing detailed traceability mappings between requirements, test cases, and functions. They propose incorporating custom domain-specific ontologies to achieve the necessary granularity and accuracy, leveraging existing knowledge graphs and relationships from sources like Java and Jira.2
